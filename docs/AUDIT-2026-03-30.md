# DogDays Codebase Audit — 2026-03-30

**Build status:** ✅ 0 warnings, 0 errors | **Tests:** ✅ all passing  
**Audited by:** Automated architecture review (4 focused sub-audits)  
**Scope:** All production source files under `src/`, all test files under `tests/`, all design documents under `docs/design/`  
**Baseline:** [AUDIT-2026-03-22](AUDIT-2026-03-22.md)

## Executive Summary

This audit covers four analysis dimensions: God class decomposition, design doc drift, guardrails compliance, and test coverage gaps. It builds on the March 22 audit baseline and tracks both resolved issues and newly identified violations.

**Issue counts:** 🔴 5 Critical | 🟠 16 Major | 🟡 12 Minor

**Key findings:**

1. **`GameplayScreen` has grown to 2,528 lines** (was 1,234 at March 22) — 3.37× the 750-line hard cap, with 25 identified responsibilities including 6 parallel arrays and 5 stringly-typed switch methods.
2. **`FishingScreen` is a second God class** at 1,975 lines (2.63× hard cap) — not flagged in the March 22 audit because it was smaller or did not yet exist.
3. **32 types are missing from design doc catalogs** — `docs/design/events.md` is completely empty despite 6 active C# events in the codebase.
4. **`TiledWorldRenderer` `SpriteBatch` leak persists** from March 22 (issue TWR-SP) — still no `IDisposable` implementation.
5. **Test coverage gaps** exist for 6 files with zero coverage and 4 files with partial coverage, including a 7-state state machine with no test coverage at all.

**March 22 resolution status:** Of the 5 critical issues from March 22, 3 have been resolved (WCM-1, IM-01, GS-08 partially) and 2 persist (IM-06 status unknown, TWR-SP confirmed unresolved). See [Resolved Since March 22](#resolved-since-march-22) for details.

## Recommended Remediation Plan

### Phase 1: Critical Fixes (standalone PRs)

1. **Fix `TiledWorldRenderer` resource leak** — Implement `IDisposable`, dispose the owned `SpriteBatch`. This is a persistent March 22 issue (TWR-SP → TWR-01).
2. **Pre-allocate `GetRenderTargets()` collections** — In both `GameplayScreen` and `FishingScreen`, replace `new List<RenderTarget2D>` returns with a pre-allocated field. `FishingScreen` calls it twice per frame.
3. **Seal `PlayerCombatStats` and `Health`** — Add `sealed` modifier per §33.

### Phase 2: GameplayScreen Decomposition (largest effort, highest payoff)

4. **Extract `EnergyOrbSystem`** (~225 lines) — Replace 6 parallel arrays with a struct-based pool. Create `EnergyOrb` struct with Position, Velocity, Timer, Tint, Type, Collected fields.
5. **Extract `ExplosionSystem`** (~60 lines) — Replace 4 parallel arrays with struct-based pool.
6. **Extract `HealthPickupSystem`** (~75 lines) — Delegate to existing `HealthPickup` entity.
7. **Extract `MapConfig` record** (~50 lines) — Replace 5 string-switch methods with a single typed record loaded from data or constructed via factory.
8. **Extract `FollowerSystem`** (~90 lines across 4 private methods) — Consolidate follower position logic.
9. **Move `CheckOcclusion` to `OcclusionRevealRenderer`** (~119 lines) — The renderer owns the occlusion render target; it should own the occlusion check logic.
10. **Extract `CrtTransitionRenderer`** (~83 lines) — CRT power transition drawing is self-contained rendering logic.
11. **Use `IWorldProp` polymorphism in `DrawWorldEntities`** (~166 lines simplified) — Iterate a single `IWorldProp[]` instead of 15+ typed arrays.

### Phase 3: FishingScreen Decomposition

12. **Extract testability seams from `FishingScreen`** — Identify and extract pure-logic helpers (state machines, coordinate conversion, spawn/cull lifecycle) into separate classes that can be tested without `GraphicsDevice`.
13. **Fix `FishingScreen.GetRenderTargets()` double-call** — Pre-allocate the render target list.

### Phase 4: IWorldProp Compliance

14. **Add `IWorldProp` to 7 missing entities** — `Firepit`, `SmallFire`, `HealthPickup`, `PlayerBlock`, `FollowerBlock`, `GnomeEnemy`, `Projectile`.

### Phase 5: Design Doc Catch-Up

15. **Populate `events.md`** with all 6 active events — include event name, declaring class, signature, and when it fires.
16. **Add 29 missing entries** across all design doc catalogs per the drift summary table.
17. **Update 2 stale entries** — `Tree` (per-species collision boxes) and `Cabin` (per-variant collision boxes).
18. **Move `FishingRippleManager`** from `rendering.md` to `systems-components.md`.

### Phase 6: Test Coverage

19. **Write `GnomeSpawnerTests.cs`** — cover spawn timing, cull logic, event emission, per-type HP assignment.
20. **Write `FlowFieldTests.cs`** — cover BFS correctness, diagonal corner-cutting, agent radius padding, unreachable tiles.
21. **Write `FishSilhouette` state machine tests** — cover all 7 states and transitions (Unaware → Curious → Approaching → StrikeReady → Spooked → Hooked → Fleeing).
22. **Extract and test `DeathScreen` timing seam** — pure-logic timing/fade state machine.
23. **Fill partial coverage gaps** — `GnomeEnemy` hit flag, `WaveManager` stagger math, `XpLevelSystem` edge cases.
24. **Fix test naming violations** — Rename 9 methods in `SlashSystemTests.cs` and 1 method in `GnomeEnemyTests.cs` to use double-underscore convention.

### Phase 7: TiledWorldRenderer Decomposition

25. **Decompose `TiledWorldRenderer`** (1,080 lines) — extract rendering passes or layer-specific logic into focused helper classes to bring the file under 750 lines.

## Guardrail Improvement Proposals

Based on the patterns found in this audit, the following improvements to [`docs/design/guardrails.md`](design/guardrails.md) are proposed. Each addresses a recurring violation pattern that the current guardrails do not explicitly prevent.

| ID | Proposal | Addresses Issues |
|---|---|---|
| **GP-A** | §30 Enforcement Consequence — PRs that introduce or modify files exceeding 751 lines must include a decomposition plan; PRs that push already-bloated files further are rejected without review. | GS-01, FS-01, TWR-01 |
| **GP-B** | §30.5 Parallel Array Anti-Pattern — Prohibit managing multiple arrays with matching indices. Require struct or class to group per-object state. | GS-02, GS-07 |
| **GP-C** | §30.6 String-Switch Map Configuration Anti-Pattern — Prohibit selecting behavior via switch expressions on asset name strings. Require typed config records. | GS-03 |
| **GP-D** | §31 Doc-Update Enforcement at PR Time — PRs that add new types without corresponding doc updates are rejected. | FD-01 through FD-08 |
| **GP-E** | §32.5 Testability Seam Requirement for GPU-Owning Screens — Game logic exceeding ~50 lines in GPU-owning screens must be extracted into testable pure-logic helper classes. | FS-01, TC-06 |
| **GP-F** | §34 Double-Underscore Enforcement — Single-underscore test method naming is explicitly prohibited. Added to §33 code review checklist. | TC-07, TC-08 |
